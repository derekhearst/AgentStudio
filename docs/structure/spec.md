# Structure Spec

## Overview

This document defines the source code organization, naming conventions, and module patterns for AgentStudio. Every contributor and agent working in this codebase should follow these conventions so that file locations are predictable and domain boundaries are clear.

## Top-Level Layout

```
src/
  app.d.ts              # SvelteKit global type augmentation (locals, session, etc.)
  app.html              # Root HTML shell
  hooks.server.ts       # SvelteKit server hooks (auth enforcement, request logging)
  lib/                  # Domain library code — shared across routes
  routes/               # SvelteKit file-based router

drizzle/                # SQL migration files (generated by drizzle-kit)
drizzle.config.ts       # Drizzle config pointing at the database
scripts/                # One-off utility scripts (not imported by the app)
docs/                   # Domain specs and plans
static/                 # Static assets served at root
src-tauri/              # Tauri desktop shell (optional build target)
```

## `src/lib/` — Domain Libraries

Each product domain lives in `src/lib/<domain>/`. The folder contains all server logic, client logic, components, and schema for that domain. No cross-domain imports except through explicit public `index.ts` exports.

### File naming conventions

| File                 | Purpose                                                                                |
| -------------------- | -------------------------------------------------------------------------------------- |
| `<domain>.schema.ts` | Drizzle table definitions and inferred TypeScript types for this domain                |
| `<domain>.server.ts` | Server-only functions (DB queries, external calls). Never imported by client bundles.  |
| `<domain>.remote.ts` | SvelteKit remote functions (`query` / `command`) — the transport layer for UI → server |
| `<domain>.ts`        | Shared types, helpers, and constants safe for both server and client                   |
| `index.ts`           | Public re-exports for the domain; only what other domains may import                   |
| `*.svelte`           | Svelte components scoped to this domain                                                |
| `*.server.test.ts`   | Server-side unit tests for this domain                                                 |
| `*.test.ts`          | Client/shared unit tests                                                               |

### The `.remote.ts` pattern

Remote functions are the only sanctioned way to call server logic from a Svelte component or client-side code. They are defined in `<domain>.remote.ts` using the `query` / `command` helpers from `src/lib/remote` (or equivalent SvelteKit RPC pattern):

- `query(...)` — read-only, can be cached, returns data
- `command(...)` — write operation, invalidates relevant queries

Components import remote functions directly:

```ts
import { listAgents } from '$lib/agents/agents.remote'
```

Direct database imports from `.server.ts` files inside components or `+page.svelte` files are not allowed.

## `src/routes/` — SvelteKit Router

Routes map directly to the URL structure. Each route folder contains the SvelteKit file conventions:

```
+page.svelte          # Page component
+page.server.ts       # Load function + form actions (server-side)
+layout.svelte        # Layout shell shared by child routes
+layout.server.ts     # Layout load function (e.g., auth check)
+error.svelte         # Error boundary for this route subtree
```

### Route → domain mapping

| Route prefix   | Domain                                             |
| -------------- | -------------------------------------------------- |
| `/chat`        | chat + runs + agents                               |
| `/agents`      | agents + skills                                    |
| `/memory`      | memory                                             |
| `/activity`    | activity                                           |
| `/skills`      | skills                                             |
| `/review`      | observability                                      |
| `/settings`    | settings + auth (passkeys)                         |
| `/users`       | auth (admin user mgmt)                             |
| `/cost`        | cost                                               |
| `/automations` | automation                                         |
| `/api/*`       | Server API routes (SSE, webhooks, cron, MCP proxy) |

### `/api/` structure

API routes handle everything that is not a page: SSE streams, webhook receivers, background cron triggers, and MCP proxy calls. They follow REST conventions (`GET` for reads, `POST` for writes). They do not use SvelteKit form actions — only JSON request/response.

```
/api/chat/[id]/stream       # SSE: live run event stream
/api/chat/[id]/approve      # POST: approve/deny tool call
/api/chat/[id]/answer       # POST: answer ask_user question
/api/agents/[id]/run        # POST: start a run for an agent
/api/mcp/[serverId]/proxy   # POST: proxy MCP tool call
/api/notifications/push     # POST: register push subscription
/api/cron/*                 # GET: cron job triggers
/api/transcribe             # POST: audio transcription
/api/upload                 # POST: file upload
```

## Schema and Migrations

All database tables are defined using Drizzle ORM. Schema files live at `src/lib/<domain>/<domain>.schema.ts`.

The `drizzle/` folder contains numbered SQL migration files generated by `bunx drizzle-kit generate`. Migrations are applied with `bunx drizzle-kit migrate`. Migration files are committed to the repo and never hand-edited after generation.

The central `src/lib/db.server.ts` exports the `db` instance used by all domain server files.

## Conventions

### Server-only code

Any file importing from the database (`db.server.ts`), using Node.js APIs, or calling external services must be named `*.server.ts` or placed in a `*.server.ts` re-export chain. SvelteKit will reject server-only imports from client bundles at build time.

### No barrel re-exports between domains

`index.ts` files expose only what other domains legitimately need. Internal helpers (e.g., `mining.server.ts` in `memory/`) are not re-exported from `index.ts`. This keeps the dependency graph clean.

### State

Client-side reactive state uses Svelte 5 runes (`$state`, `$derived`, `$effect`). Global client state (e.g., current session, active run) lives in `src/lib/state.svelte.ts`. Components do not hold cross-component state in module-level variables.

### Components

Svelte components live in the domain folder they belong to. Shared/generic components (buttons, modals, form inputs, layout primitives) live in `src/lib/ui/`. Components are PascalCase. Files that are not components are camelCase.

### Error handling

Server functions throw typed errors using a shared `AppError` class or SvelteKit's `error()` helper. Remote functions catch and re-throw as structured responses. Components receive error state via remote function return values, not exception propagation.
